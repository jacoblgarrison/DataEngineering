End-to-End Analytics Pipeline

Data Source:
Public transactional dataset representing customer purchases (CSV files ingested into PostgreSQL).

Problem:
Raw business data was difficult to analyze due to inconsistent schemas and lack of analytics-ready tables.

What I Built:

  • Designed a star schema for analytics
  • Built SQL transformations from raw to analytics layers
  • Created reporting-ready fact and dimension tables

Architecture:

  • Raw data ingested into staging tables
  • SQL transformations applied to clean and standardize data
  • Analytics layer modeled using fact and dimension tables
  • Final tables exposed for reporting and downstream analytics

Tech Stack:

  • PostgreSQL
  • SQL
  • Python (Pandas)
  • Git / GitHub

Key Engineering Decisions:

  • Used SQL-first transformations for transparency and maintainability
  • Modeled data using a star schema to optimize analytical queries
  • Separated raw, staging, and analytics layers to reduce coupling
  • Designed tables to support incremental data loads

Advanced SQL Analytics System

Problem:
Business metrics required complex logic that was difficult to maintain across dashboards and ad hoc queries.

Data Source:
Relational tables representing transactions, users, and events stored in PostgreSQL.

What I Built:

  • Centralized metric definitions using SQL views
  • Implemented window functions for time-based analysis
  • Created reusable CTEs to simplify complex logic

Key Engineering Decisions:

  • Encapsulated business logic in SQL to reduce dashboard complexity
  • Optimized queries to minimize compute and execution time
  • Designed queries for readability and long-term maintainability

Tech Stack:

  • PostgreSQL
  • SQL (CTEs, window functions)
